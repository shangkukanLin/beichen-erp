# 北辰公司屏幕 ERP 系统技术选型方案

> 版本：v1.0 ｜ 适用范围：Web 端 + 微信小程序端 ｜ 业务域：研发管理 / 委外加工 / 进销存 / 财务一体化 / 智能体辅助

---

## 一、业务链路与系统边界梳理

在选型之前，先把北辰公司的核心业务链路拆成可被系统承接的模块，技术方案必须围绕这条链路闭环设计：

```
立项研发 ──► BOM 搭建 ──► 采购计划 ──► 物料收发 ──► 委外加工
   │             │            │            │            │
   ▼             ▼            ▼            ▼            ▼
研发项目管理   BOM/图纸库   采购订单     库存管理     加工单/工单
(文档/周期/BUG)            (供应商)    (LED玻璃/辅料) (加工费/损耗/时效)
                                                          │
   ┌──────────────────────────────────────────────────────┘
   ▼
委外成品入库 / 第三方采购成品入库 ──► 成品进销存 ──► 销售订单 ──► 财务管理
                                       │              │            │
                                       ▼              ▼            ▼
                                    成品库存       客户/报价     应收应付/资金
                                                              (按时间生成账单)
```

由此可提炼出 **8 大业务模块 + 1 个智能体层**：

| 模块 | 核心诉求 | 系统特征 |
|------|---------|---------|
| 研发项目管理 | 排线图纸/BOM/开发周期/BUG 列表 | 文档管理、流程状态机、版本控制 |
| BOM 与物料主数据 | 物料编码、BOM 多级、替代料 | 树形结构、强一致性、主数据中心 |
| 采购计划与订单 | 按BOM/库存自动测算、供应商管理 | MRPA 逻辑、审批流 |
| 仓储与收发存 | LED玻璃原材/辅料/半成品/成品多仓 | 批次/序列号、先进先出、实时库存 |
| 委外加工 | 加工单、物料损耗、加工费、时效 | 工单状态机、成本归集 |
| 销售与报价 | 客户、报价、销售订单、发货 | 价格策略、毛利测算 |
| 财务一体化 | 应收应付、资金、按期账单 | 账期引擎、对账、凭证 |
| 智能体层 | 自动记录、查询、成本预警 | LLM + RAG + 工具调用 |

## 二、整体架构设计

采用 **前后端分离 + 微服务化后端 + 独立 AI 服务** 的分层架构：

```
┌─────────────────────────────────────────────────────────┐
│                      接入层                              │
│   Web 端 (PC)          微信小程序端        开放 API       │
│  Vue3+ElementPlus     uni-app             (未来对接)     │
└──────────────┬──────────────┬──────────────┬────────────┘
               │              │              │
        ┌──────▼──────────────▼──────────────▼──────┐
        │            API 网关 (鉴权/限流/路由)        │
        └──────┬─────────────────────────────────────┘
               │
   ┌───────────┼───────────┬───────────┬───────────┐
   ▼           ▼           ▼           ▼           ▼
 研发服务    供应链服务   仓储服务    销售服务    财务服务
 (Java)      (Java)       (Java)      (Java)      (Java)
   │           │           │           │           │
   └───────────┴─────┬─────┴───────────┴───────────┘
                     │
        ┌────────────▼────────────┐       ┌─────────────────┐
        │   智能体服务 (Python)    │◄──────► 本地大模型+向量库 │
        │   自动记录/查询/预警     │       │ (RAG/Function)  │
        └─────────────────────────┘       └─────────────────┘
                     │
   ┌─────────────────┴──────────────────┐
   ▼                ▼                   ▼
 MySQL(主)      Redis(缓存)        对象存储(MinIO 自建)
 PostgreSQL     消息队列            Elasticsearch(检索)
 (文档/日志)
```

**设计原则：**
- **主业务用 Java 微服务**：保证事务一致性、审计合规、长期可维护（ERP 的命脉）。
- **智能体用 Python 独立服务**：复用 AI 生态，与主业务解耦，避免互相拖累。
- **统��网关**：Web、小程序、AI 服务统一鉴权与路由。
- **小程序只做轻量操作**：查询、审批、扫码出入库、看板；重录入留在 Web 端。

## 三、技术选型详解

### 3.1 后端主业务服务

| 候选方案 | 优势 | 劣势 | 适配度 |
|---------|------|------|--------|
| **Java + Spring Boot 3** *(推荐)* | 企业级ERP事实标准；事务/权限/审批生态成熟；人才好招；长期维护有保障 | 启动较慢、代码量偏大 | ★★★★★ |
| Node.js + NestJS | 前后端同语言；开发快 | 复杂事务/财务精度生态弱；ERP实践少 | ★★★ |
| Python + Django | AI友好、开发快 | 高并发与复杂事务弱；性能不如JVM | ★★★ |
| Go + go-zero | 性能极佳、部署轻 | ERP业务生态薄、ORM/工作流弱 | ★★ |

**推荐：Java 17 + Spring Boot 3 + Spring Cloud Alibaba**

关键组件：
- **持久层**：MyBatis-Plus（国内 ERP 主流，动态 SQL 灵活）
- **权限**：Sa-Token（轻量、国产、文档好）或 Spring Security
- **工作流/审批**：Flowable（BPMN 标准，适合采购审批、账单审核）
- **任务调度**：XXL-Job（账单生成、库存盘点、数据同步）
- **API 文档**：SpringDoc (OpenAPI 3)
- **数据导出**：EasyExcel（BOM、账单、报表必备）

> 选 Java 的核心理由：ERP 的财务核算、库存账实、审批留痕要求**强事务、强审计、强类型**，Java 生态在这三点的成熟度远超其他选项，且 5 年后仍易招人维护。

### 3.2 智能体服务（AI 层）

独立 Python 微服务，通过网关与主业务交互：

- **框架**：FastAPI（异步、轻量、自带文档）
- **Agent 编排**：LangChain / LangGraph（工具调用、多步推理）
- **LLM 接入**：**本地部署开源大模型**，不依赖任何云端 API。推荐方案：
  - **推理框架**：Ollama（轻量易用，适合中小规模）或 vLLM（高吞吐，生产级）
  - **推荐模型**：Qwen2.5-7B/14B（通义千问开源版，中文能力强）、GLM-4-9B（智谱开源）、DeepSeek-V2-Lite（开源权重可本地跑）
  - **选型依据**：均为开源权重、可完全离线运行、中文理解优秀；模型权重下载到本地服务器后断网也可推理
  - **硬件要求**：7B 模型约需 16GB 显存（单张 RTX 4090/A10 可跑），14B 约 32GB；若服务器无 GPU，可用 CPU 推理（Ollama 支持），但响应较慢，适合低频查询场景
- **向量检索 RAG**：业务文档（BOM说明、图纸备注、操作日志）向量化，支持"问库存、问订单状态、问某项目进度"
- **Function Calling**：把"查库存、生成日报、对账、预警缺料"封装成工具��由 Agent 自主调用

**智能体典型场景：**
1. 语音/文字录入"今天收到友达 500 片玻璃"→ 自动生成入库单草稿
2. "查一下 A 项目当前进度和未关闭 BUG"→ 汇总返回
3. 每日自动生成加工损耗与成本日报
4. 物料低于安全库存时主动推送预警

### 3.3 Web 端前端

| 候选方案 | 优势 | 劣势 | 适配度 |
|---------|------|------|--------|
| **Vue 3 + Element Plus** *(推荐)* | 中后台生态最全；表格/表单/树组件强；国内文档好 | — | ★★★★★ |
| React + Ant Design | 组件丰富、社区大 | 学习曲线略高；ERP 国内 Vue 占多 | ★★★★ |
| Vue 3 + Ant Design Vue | 组件质量高 | ERP 场景表格不如 ElementPlus 顺手 | ★★★★ |

**推荐：Vue 3 + TypeScript + Element Plus + Vite + Pinia**

关键组件：
- **表格**：VxeTable（虚拟滚动、单元格编辑，BOM/账单必备）或 Element Plus Table
- **图表**：ECharts（库存看板、成本趋势、毛利分析）
- **表单/校验**：Element Plus Form + 自定义校验
- **状态管理**：Pinia
- **请求**：Axios + 统一拦截（鉴权、错误、loading）
- **权限**：动态路由 + 按钮级指令 `v-permission`

### 3.4 小程序端

| 候选方案 | 优势 | 劣势 | 适配度 |
|---------|------|------|--------|
| **uni-app (Vue3)** *(推荐)* | 一套代码出微信小程序/H5/App；Vue 技能复用；生态全 | 个别原生能力需适配 | ★★★★★ |
| 微信原生小程序 | 性能最优、能力最新 | 仅限微信；无法复用 Web 技能 | ★★★★ |
| Taro (React) | 多端、React 生态 | 团队若用 Vue 则不划算 | ★★★ |

**推荐：uni-app + Vue 3 + uView UI / uni-ui**

小程序定位为**移动轻端**，聚焦以下场景：
- 扫码出入库（物料/成品条码）
- 审批待办（采购单、加工单、账单）
- 库存与订单实时查询
- 数据看板（管理层）
- 智能体对话入口（语音/文字录入、问答）

### 3.5 数据库与存储

| 用途 | 选型 | 说明 |
|------|------|------|
| **核心业务库** | MySQL 8.0 *(主)* | 事务强、ERP 主流；财务/库存/订单主数据 |
| 文档/日志/审计 | PostgreSQL *(可选)* | JSONB 存研发文档元数据、操作流水 |
| 缓存/会话 | Redis 7 | 鉴权 token、库存热数据、限流 |
| 全文检索 | Elasticsearch | BOM/物料/订单检索、智能体检索增强 |
| 对象存储 | **MinIO** *(自建私有化部署)* | 排线图纸、BOM 表、合同、发票附件；S3 协议兼容，部署在自有云服务器，数据完全自主可控 |
| 消息队列 | RabbitMQ | 库存变更广播、账单生成、AI 任务异步化 |
| 向量库 | Milvus / pgvector | 智能体 RAG 检索 |

> **财务精度红线**：金额一律用 `DECIMAL(18,4)`，严禁浮点；所有库存、成本变动必须可追溯（流水表 + 单据）。

> **MinIO 自建部署要点**：Docker 一键部署到自有云服务器，S3 协议兼容（应用层用 AWS S3 SDK 即可对接，未来无需改代码）；建议挂载独立数据盘并配置定时快照；单节点起步，数据量增大后可平滑扩展为分布式集群；图纸/BOM 等文件按业务模块分 Bucket（如 `rd-drawings`、`bom-files`、`contracts`、`invoices`），配合访问策略隔离权限。

### 3.6 基础设施与部署

- **容器化**：Docker + Docker Compose（中小规模）/ K8s（规模化）
- **CI/CD**：**Jenkins**（自建，开源）或 GitLab CE（自建私有代码仓库 + CI/CD 一体）
- **监控**：Prometheus + Grafana（系统）、SkyWalking（链路追踪）
- **日志**：ELK / Loki
- **私有化部署**：全部组件自建在自有云服务器，数据不出企业，**不依赖任何托管服务**（含大模型、对象存储、CI/CD、代码仓库）

## 四、推荐方案总览

```
┌─ Web 端 ──────────── Vue3 + TS + Element Plus + Vite + Pinia + ECharts
├─ 小程序端 ────────── uni-app (Vue3) + uView UI
├─ API 网关 ────────── Spring Cloud Gateway (鉴权/限流/路由)
├─ 主业务服务 ──────── Java 17 + Spring Boot 3 + MyBatis-Plus + Sa-Token
│                       + Flowable(审批) + XXL-Job(调度) + EasyExcel
├─ 智能体服务 ──────── Python + FastAPI + LangChain + 向量库
│                       + 本地大模型 (Ollama/vLLM + Qwen2.5/GLM-4)
├─ 数据库 ──────────── MySQL 8 (主) + Redis + Elasticsearch
├─ 文件存储 ────────── MinIO (自建，S3 协议兼容)
├─ 中间件 ──────────── RabbitMQ
├─ 代码仓库/CI/CD ──── GitLab CE 或 Jenkins (自建)
└─ 部署 ───────────── Docker + Compose / K8s + Prometheus + Grafana
```

**为何这样组合（一句话总结）：**
> 主业务用 Java 求稳求合规，智能体用 Python 求快求智能，前端 Vue3 一套技能打通 Web 与小程序，MySQL+Redis 保证账实准确，整体可私有化、可扩展、5 年内不踩坑。

## 五、关于"是否引入工作流引擎"的取舍

北辰的业务里有大量审批（采购单、加工单、账单、出库），建议**引入 Flowable**：
- 优点：审批流可视化配置、留痕、可审计；账单审核流程可随业务调整而无需改代码。
- 代价：学习成本。若初期审批简单（1~2 级线性），可先用状态字段 + 代码硬编码，后期再升级到 Flowable，避免过度设计。

## 六、开发节奏建议（分三期）

| 阶段 | 周期 | 目标模块 | 交付价值 |
|------|------|---------|---------|
| **一期：底座+进销存** | 2~3 月 | 物料主数据/BOM/采购/仓储/销售/基础财务 | 跑通物料→库存→销售闭环 |
| **二期：委外+研发** | 2~3 月 | 委外加工/研发项目/文档库/成本核算 | 打通研发与生产全链路 |
| **三期：财务+智能体** | 2 月 | 应收应付/账单引擎/资金/智能体 | 财务一体化 + AI 辅助上线 |

> 智能体不必等三期，可在一期就做"问答查询"轻量版，三期再补"自动录入/预警"。

## 七、风险提示

1. **BOM 多级与成本归集** 是屏幕行业 ERP 最易翻车点，建议一期就把 BOM 结构与成本归集模型设计清楚，后期改动成本极高。
2. **委外加工的损耗与时效** 需在工单模型里预留"标准用量/实际用量/损耗率/加工周期"字段，否则后期核算补不上。
3. **财务账期** 要支持"按月/按周/按订单"多种账单生成策略，避免写死。
4. **小程序性能**：库存查询、看板数据要走 Redis 缓存，避免直接打数据库。
5. **智能体幻觉**：涉及金额/库存等关键数据，Agent 必须走 Function Calling 调真实接口，不允许 LLM 自由生成数字。
6. **本地大模型硬件**：若服务器无独立 GPU，7B 模型 CPU 推理单次响应约 3~10 秒，仅适合低频查询；高频使用建议配置一张 RTX 4090 或 A10 显卡。
